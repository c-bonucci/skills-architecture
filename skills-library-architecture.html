01

Overview

The Skills Library is a centralised repository of reusable resources — starting with skills, expanding later to tools, templates, and code components. Each resource is a folder of files (scripts, config, instructions) stored in a Git repository. The platform reads from git on demand, caches content at link time, and never fetches from git during pipeline execution.

Customer-owned repositories can be connected; they surface alongside platform skills in the library UI and follow the same link-and-cache mechanic. Skills can also be served externally to other services via a stable API using SHA references.

Library

Git repo as source of truth. Resources browsed on-demand via provider API. Version history visible. No data stored locally except cached snapshots.

Pipeline Linking

At link time: fetch from git, cache content, store SHA reference in pipeline definition. At run time: read from cache only. Zero runtime dependency on git.

External Serving

API endpoint serves skill content by name + SHA. Consumers can pin to a specific version or request latest. Content is served from cache, not git directly.

02

Data Model

These are the core entities stored in the platform database (Postgres/RDS). Git content itself is never stored in full — only metadata and cached snapshots.

Entity Key Fields Notes
repo_connection id, org_id, provider (github|gitlab), repo_url, encrypted_token, token_type (pat|oauth|app), created_at One per connected repo. Token stored encrypted via KMS. Never returned in API responses.
skill_cache id, repo_connection_id, skill_path, sha, files (jsonb), cached_at, expires_at Immutable once written. Keyed by repo + path + SHA. Content stored as JSON map of filename → base64 content.
pipeline_skill_ref id, pipeline_id, skill_name, repo_connection_id, skill_path, pinned_sha, linked_at Created at link time. Stores the SHA pinned at that moment. Backend resolves the cache entry at runtime using repo + path + sha — no cache_id stored in the YAML or ref.

Pipeline YAML — skill reference format

The pipeline definition stays flat. No resources/ folder. The YAML stores only the path and pinned SHA — the backend resolves the cache entry from those two values at runtime. No internal IDs leak into the pipeline definition.

name: my-pipeline
version: 1.0.0

skills:
  - name: skill_a
    source: platform          # or "repo" for customer-connected repos
    path: skills/skill_a
    sha: a3f9c1d             # pinned at link time, never changes

  - name: cust_skill_x
    source: repo
    repo_connection_id: rc_09az...
    path: skills/cust_skill_x
    sha: 7d4c91f
03

Data Flow

There are four distinct flows. Git is only involved in flows 1 and 2. Flows 3 and 4 read from cache only.

Flow A — Browse the library

Flow B — Link a skill to a pipeline

Flow C — Run a pipeline

Flow D — Serve skill to external service (e.g. Claude)

API Endpoints

Library & Browsing

GET
/api/library
Returns merged skill list from platform repo + any connected customer repos. Calls git provider on demand, no DB read.
GET
/api/library/:skill/history
Returns commit list for a skill folder. Maps to GET /repos/.../commits?path= on the provider.
GET
/api/library/:skill/versions/:sha
Returns skill file contents at a specific SHA. Used for preview before linking.

Repo Connections

POST
/api/repos/connect
Saves a new repo connection. Encrypts token via KMS before writing to DB. Validates access immediately.
GET
/api/repos
Lists connected repos for the org. Never returns token values.
DEL
/api/repos/:id
Removes connection. Does not delete cached content — existing pipeline refs remain valid.

Pipeline Linking

POST
/api/pipelines/:id/skills/link
Fetches skill at HEAD (or specified SHA), writes to skill_cache, creates pipeline_skill_ref. Body: { skill_path, repo_connection_id?, sha? }
POST
/api/pipelines/:id/skills/:ref/update
Re-fetches skill at latest HEAD, creates new cache entry, updates the pipeline_skill_ref SHA. Explicit user action only.
DEL
/api/pipelines/:id/skills/:ref
Removes skill reference from pipeline. Cache entry is retained (may be shared with other pipelines).

External Serving

GET
/api/serve/:skill_name@:sha
Returns skill content at exact SHA from cache. Requires API key auth. Returns 404 if SHA not cached (consumer must link first).
GET
/api/serve/:skill_name@latest
Resolves HEAD SHA, fetches from cache (or git if not cached), returns content. Consumer accepts that this may change.
05

Git Provider Abstraction

All git operations go through a single GitProvider interface. The rest of the backend never calls a provider API directly. Adding a new provider (Bitbucket, self-hosted Gitea) is a new implementation of this interface only.

// interface — provider-agnostic
interface GitProvider {
  listSkills(repoUrl: string, path: string): Promise<SkillEntry[]>
  getFiles(repoUrl: string, path: string, sha: string): Promise<FileMap>
  getHistory(repoUrl: string, path: string): Promise<Commit[]>
  resolveHead(repoUrl: string, branch: string): Promise<string> // returns SHA
  createCommit(repoUrl: string, path: string, files: FileMap, message: string): Promise<string>
}

// implementations
class GitHubProvider implements GitProvider { /* maps to GitHub REST API */ }
class GitLabProvider implements GitProvider { /* maps to GitLab REST API */ }

// factory — resolves provider from stored repo_connection
function getProvider(connection: RepoConnection): GitProvider {
  const token = kms.decrypt(connection.encrypted_token)
  if (connection.provider === 'github') return new GitHubProvider(token)
  if (connection.provider === 'gitlab') return new GitLabProvider(token)
  throw new Error(`Unknown provider: ${connection.provider}`)
}

GitHub implementation

Maps each interface method to a GitHub REST endpoint. Auth via PAT (v1) or GitHub App short-lived tokens (v2). Self-hosted GitHub Enterprise supported by swapping the base URL.

// listSkills → GET /repos/{owner}/{repo}/contents/{path}
// getFiles   → GET /repos/{owner}/{repo}/contents/{path}?ref={sha}
// getHistory → GET /repos/{owner}/{repo}/commits?path={path}
// resolveHead→ GET /repos/{owner}/{repo}/git/ref/heads/{branch}
// createCommit→ PUT /repos/{owner}/{repo}/contents/{path}

GitLab implementation

Maps to GitLab REST v4 API. Supports gitlab.com and self-hosted instances — base URL is configurable per repo_connection. Auth via OAuth token or PAT.

// listSkills → GET /projects/{id}/repository/tree?path={path}
// getFiles   → GET /projects/{id}/repository/tree?path={path}&ref={sha}
// getHistory → GET /projects/{id}/repository/commits?path={path}
// resolveHead→ GET /projects/{id}/repository/branches/{branch}
// createCommit→ POST /projects/{id}/repository/commits

Caching Strategy

Caching is essential to this architecture — it is what decouples pipeline execution from any runtime dependency on git. The exact implementation (storage backend, eviction policy, cache key structure) is TBD and will be worked out as a separate exercise.

What must be cached

The full file contents of a skill folder at a specific SHA. Once written, this entry is immutable — a SHA in git never changes, so the content never changes. It must be keyed by at minimum (repo, skill_path, sha).

When it must be populated

At link time — the cache entry must exist before the pipeline_skill_ref is created. At runtime the backend looks up the cache using the repo + path + sha from the pipeline YAML. If the entry is missing, the run must fail loudly rather than fall back to git.

Library listing cache

The skill folder listing (names, metadata) fetched from git on library page load should be short-lived cached to avoid hitting provider rate limits on every request. Exact TTL and storage TBD.

Open questions

Where does cache content live — same DB, S3, Redis? How do we handle cache size growth over time? What happens on a cache miss at runtime — hard fail or fallback? Cleanup policy for entries with no live pipeline refs.

07

Pipeline Linking — Detailed Flow

This is the most critical operation. It must be atomic — either the cache entry and the pipeline ref are both created, or neither is. A partial link leaves the pipeline in an inconsistent state.

// POST /api/pipelines/:id/skills/link
async function linkSkill({ pipelineId, skillPath, repoConnectionId, sha? }) {

  // 1. Resolve SHA — use provided SHA or fetch HEAD
  const connection = await db.repoConnections.get(repoConnectionId)
  const provider  = getProvider(connection)
  const resolvedSha = sha ?? await provider.resolveHead(connection.repoUrl, 'main')

  // 2. Populate cache — skip fetch if already cached at this SHA
  // (exact cache implementation TBD)
  const cached = await cache.get(repoConnectionId, skillPath, resolvedSha)
  if (!cached) {
    const files = await provider.getFiles(connection.repoUrl, skillPath, resolvedSha)
    await cache.set(repoConnectionId, skillPath, resolvedSha, files)
  }

  // 3. Create pipeline ref — repo + path + sha is enough for backend to resolve cache
  await db.pipelineSkillRefs.create({
    pipelineId, skillPath,
    repoConnectionId, pinnedSha: resolvedSha
  })

  // 4. Update pipeline.yaml — path + sha only, no internal IDs
  await pipelines.addSkillRef(pipelineId, { skillPath, sha: resolvedSha })
}

Updating a linked skill

Updating is an explicit user action — never automatic. The user sees the current pinned SHA and the latest SHA in the UI. If they differ, an "Update available" badge appears. Clicking it triggers the same link flow with the new HEAD SHA, creating a new cache entry and updating the pipeline ref.

Old cache entries are retained — other pipelines may still reference them. Cleanup of unreferenced cache entries can be a background job run periodically.

08

External Serving

External services (e.g. Claude, partner integrations) request skill content via a public API. All responses come from cache — git is never called during serving.

Pinned version

# Always returns the same content
GET /api/serve/skill_a@a3f9c1d
Authorization: Bearer <api_key>

# Response
{
  "skill": "skill_a",
  "sha": "a3f9c1d",
  "files": {
    "run.py": "...",
    "config.json": "..."
  }
}

Latest version

# Resolves HEAD at request time
GET /api/serve/skill_a@latest
Authorization: Bearer <api_key>

# Response includes resolved SHA
{
  "skill": "skill_a",
  "sha": "b18e2a4",   // actual SHA
  "files": { "..." }
}
# Consumer can pin to this SHA for stability

Risks & Mitigations

Risk Likelihood Impact Mitigation
Git provider unreachable during browse Medium Low Show last-known library list from short-lived cache. Pipeline execution unaffected.
Token leak / credential exposure Low High KMS encryption at rest. Never return tokens in API responses. GitHub App tokens expire in 1 hour. Audit log all decryption events.
Git provider API rate limits Medium Medium Short-lived library listing cache (60s). Link operations are infrequent. GitHub App has 15,000 req/hour vs PAT's 5,000.
skill_cache grows unbounded Low Low Background job periodically removes cache entries with no live pipeline_skill_ref. Skills are small files — growth is slow.
Customer deletes connected repo Low Medium Existing cached content and pipeline refs are unaffected. Library browse for that repo shows an error state. Notify org admins.
