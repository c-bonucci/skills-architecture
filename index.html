Architecture Decision

Resources Library — Storage & Versioning Model

We're starting with skills, but this model will set the foundation for the full resources library — tools, templates, code components, and more. The architecture we pick now needs to scale to all of those.

Overview

Skills live in S3/RDS. No versioning — every edit replaces "latest." When a user links a skill to an agent, the platform copies the entire skill folder into that pipeline's resources/ folder. The YAML references skills locally from there, and the whole unit (yaml + resources/) versions together. Edits via the UI update only that pipeline's copy — other pipelines are unaffected.

Customer repos: once connected, the library page does a live read-only pull to show the latest skills from that repo alongside platform skills. They're browseable but not editable in the UI. Linking any of them triggers the same copy event — the folder lands in resources/ just like a platform skill would.

Scope: starting with skills, but the same copy-on-link model would apply to tools, templates, code components, and other resource types as the library grows.

Advantages

Drawbacks

  • Bug fix in a platform skill must be re-linked or manually patched in every copy
  • Storage grows linearly with the number of links made

Serving Skills to External Services
e.g. linking a skill from the library directly to Claude or another platform

An external service points at a skill from the library, not from a pipeline copy. The backend exposes an API endpoint reading from S3/RDS and serving the skill folder contents. Since there is no version history, every consumer always gets the current "latest" — there is no way to pin to a specific version. If someone edits the skill, the external consumer silently receives the updated version with no warning and no rollback path.

Enabling version pinning would require building versioning separately on top of S3/RDS (e.g. timestamped snapshots), adding significant complexity that isn't native to this model.

Advantages
  • Simple to expose — an API that reads from S3/RDS is straightforward to build
  • External consumers always get the latest automatically, no action needed
Drawbacks
  • No stable reference — external consumers cannot pin to a specific version
  • Silent updates — a skill edit immediately affects all external consumers
  • No rollback if a bad update propagates to an external service
4
Self-Learning Skills
Agents identify common patterns and automatically propose or create skills

An agent writes a new skill folder directly to S3/RDS — a straightforward API call, no commit or PR needed. The skill appears in the library immediately as "latest" alongside human-curated skills. There is no built-in distinction between auto-generated and human-authored skills, and no creation history.

At scale a separate tagging or approval layer would be needed to avoid cluttering the library — this logic must be custom-built, as the model provides no native governance mechanism.

Advantages
  • Very easy to implement — agents write to S3/RDS via the existing backend
  • Skills surface in the library immediately, no intermediate step
Drawbacks
  • No authorship or provenance — no way to tell what created a skill or when
  • No approval gate — auto-generated skills land directly in the library
  • Governance (review, approval, promotion) must be built from scratch
Overview

Skills live in a Git repository (platform-owned or customer-owned). The Resources page shows the latest commit, but users can browse any prior version by commit SHA. Linking a skill to an agent stores only a reference: file path + SHA — no duplication, and no change to the pipeline folder structure. The pipeline.yaml stays flat, pointing to an external ref rather than a local copy. Whether edits can happen from the product UI is still an open question (they could create commits, or we could require repo-side changes only).

Scope: starting with skills, but the git-backed model would extend naturally to tools, templates, code components, and other resource types — each a versioned file or folder in the same repo.

❓

Customer repo topology: one repo per customer (clean isolation, more repos to manage) or one shared repo with subfolders (simpler ops, but access control per subfolder is harder to enforce cleanly)?

❓

UI editing: do we let users edit in-product and auto-commit, or keep the repo as the only write path? Big UX and complexity tradeoff.

❓

Platform repo provider: self-hosted Gitea, GitHub-backed, or something else? Affects cost, maintenance, and customer SSO/auth story.

3
Serving Skills to External Services
e.g. linking a skill from the library directly to Claude or another platform

An external service points at a skill from the library using a stable reference: skill_a@a3f9c1d. The SHA is an immutable identifier — the external consumer is guaranteed to always receive the exact same artifact regardless of what happens to the skill later. You can also expose skill_a@latest as a moving pointer for consumers who want to always get the newest version.

This is the same model used by npm, PyPI, and HuggingFace. The git repo becomes a lightweight registry with no extra infrastructure needed beyond the serving API.

Advantages
  • Stable, immutable references — external consumers can pin to an exact SHA
  • Choice of pinned or latest — consumer decides how much stability they need
  • No extra versioning infrastructure — git history is the registry
Drawbacks
  • External consumers on a pinned SHA won't receive bug fixes automatically
  • Serving API needs to resolve SHAs and stream folder contents from git
4
Self-Learning Skills
Agents identify common patterns and automatically propose or create skills

An agent creates a new skill by opening a commit — or a branch and PR — in the git repo. This gives every auto-generated skill full authorship, a timestamp, a diff, and a complete trail of what changed. A human approval step can be enforced before the skill is merged and visible in the library, keeping auto-generated and curated skills clearly separated.

The write path is more complex than a direct S3 write, but the governance comes for free from git's native model — no custom approval infrastructure needed.

Advantages
  • Full provenance — every auto-generated skill has authorship, timestamp, and diff
  • Native approval gate — PR review before a skill is promoted to the library
  • Auto-generated and human-authored skills are naturally distinguishable
Drawbacks
  • More complex write path — agents need to commit to a repo, not just write a file
  • PR-based approval adds latency before a skill is usable
